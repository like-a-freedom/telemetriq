# Plan: Fix Telemetry Overlay Lag — Real-Time Pace Responsiveness

## Problem
Overlay metrics (especially pace) lag behind video by up to 4 seconds. A runner transitioning from 5:30/km pace to walking still shows the fast pace for several seconds.

## Root Cause

`interpolatePace` (`telemetryCore.ts:947-999`) is the **sole display-time lag source**:

1. **Snaps to integer second** (line 956): `Math.round(gpxTime)` — loses sub-second precision
2. **Symmetric ±4s median** (lines 958, 967): collects ALL frames within 4 seconds on BOTH sides
3. **Returns median of that window** — when athlete decelerates, the median is dominated by prior faster values for up to 4 seconds

Other components are NOT lag sources:
- `stabilizeSegmentSpeeds` (build-time, line 414): only replaces isolated spikes, preserves sustained changes via sustained-change guard (lines 461-463)
- `interpolateTelemetryMetric` (line 914): simple lerp, no median — speed/grade display correctly
- `fillMissingMetricValues` (build-time, line 486): pure linear interpolation, no median (the old `fillMissingPaceValues` with window=7 is dead code — not called in current pipeline)
- `MEDIAN_SPEED_WINDOW=5` (line 26): only used by `computeMedianPace` which is exported but NOT called in the main pipeline

## Approach

Replace the symmetric ±4s median in `interpolatePace` with **lerp-first, small backward-biased median fallback**.

### Why lerp-first is safe

The build-time spike filter (`stabilizeSegmentSpeeds`) already handles heavy spike rejection with ±2 neighbor radius and 35% deviation threshold. By the time frames reach display time, they are already clean. The display-time `interpolatePace` is a secondary safety net, not the primary filter.

### Changes

#### 1. `frontend/src/modules/telemetryCore.ts` — `interpolatePace` (line 947-999)

Replace with:

```typescript
function interpolatePace(
    frames: TelemetryFrame[],
    gpxTime: number,
): number | undefined {
    const { beforeFrame, afterFrame, interpolationFactor } = findSurroundingFrames(frames, gpxTime);
    const frameGapSeconds = afterFrame.timeOffset - beforeFrame.timeOffset;

    // Dense sampling (≤2s gap): lerp between surrounding frames (causal, no lag)
    if (frameGapSeconds <= MAX_INTERPOLATED_PACE_GAP_SECONDS) {
        const directPace = interpolateOptionalValue(
            beforeFrame.paceSecondsPerKm,
            afterFrame.paceSecondsPerKm,
            interpolationFactor,
        );
        if (directPace !== undefined) return directPace;

        // Fallback: median of ±1s neighborhood for spike resistance when lerp fails
        const nearbyPaces = frames
            .filter((frame) => Math.abs(frame.timeOffset - gpxTime) <= 1 && frame.paceSecondsPerKm !== undefined)
            .map((frame) => frame.paceSecondsPerKm as number);
        if (nearbyPaces.length >= 2) {
            return median(nearbyPaces);
        }

        // Tertiary: speed-based pace from ±1s neighborhood
        const nearbySpeedMs = frames
            .filter((frame) => Math.abs(frame.timeOffset - gpxTime) <= 1 && frame.speedKmh !== undefined)
            .map((frame) => frame.speedKmh! / 3.6);
        const nearbySpeedMedian = median(nearbySpeedMs);
        if (nearbySpeedMedian !== undefined) {
            return speedMsToPaceSecondsPerKm(nearbySpeedMedian);
        }
    }

    // Sparse gaps (2-5s): hold last known pace to avoid inventing ramps
    if (frameGapSeconds > MAX_INTERPOLATED_PACE_GAP_SECONDS
        && frameGapSeconds <= MAX_CONSECUTIVE_GAP_SECONDS) {
        return beforeFrame.paceSecondsPerKm ?? afterFrame.paceSecondsPerKm;
    }

    // Very sparse gaps (>5s): hold last known pace
    if (frameGapSeconds > MAX_CONSECUTIVE_GAP_SECONDS) {
        return beforeFrame.paceSecondsPerKm ?? afterFrame.paceSecondsPerKm;
    }

    // Fallback: interpolate
    return interpolateOptionalValue(
        beforeFrame.paceSecondsPerKm,
        afterFrame.paceSecondsPerKm,
        interpolationFactor,
    );
}
```

**Key differences from current code:**
| Aspect | Current | Proposed |
|--------|---------|----------|
| Time snap | `Math.round(gpxTime)` — integer second | Exact `gpxTime` — sub-second precision |
| Neighborhood | ±4 seconds symmetric | ±1 second (only for spike fallback) |
| Primary algorithm | Median of ±4s window | Linear interpolation (causal, responsive) |
| Median role | Primary path | Fallback when lerp fails (undefined values) |
| Speed-based median | ±4s window, tried first | ±1s window, tried last |
| Lag introduced | Up to 4 seconds | Near-zero (lerp is instantaneous) |

**Edge cases handled:**
- **Fast deceleration**: lerp transitions pace over 1 frame gap (correct — pace IS transitioning)
- **Single-frame spike**: ±1s median catches it (3-frame window is sufficient)
- **Sparse 3-5s GPX**: hold behavior unchanged (correct — don't interpolate across gaps)
- **Very sparse >5s GPX**: hold behavior unchanged
- **GPX with 1s sampling, both frames have pace**: lerp returns immediately (fast path)
- **GPX with 1s sampling, one frame lacks pace**: falls to ±1s median or speed median

#### 2. `frontend/src/__tests__/telemetryCore.test.ts` — New tests

Add `describe('interpolatePace responsiveness')`:

1. **Deceleration responsiveness**: Create 10 frames with pace transitioning from 300s/km (5:00) to 600s/km (10:00) at frame index 5. Verify `getTelemetryAtTime` at frame 6 shows pace closer to 600 than 300 (within 1 second of transition, not 4 seconds).

2. **Acceleration responsiveness**: Same in reverse — slow to fast. Verify pace drops quickly.

3. **Single spike rejection**: Frames at consistent 300s/km with one outlier at 900s/km. Verify the outlier is smoothed (lerp between neighbors, or ±1s median catches it).

4. **Sparse gap hold**: Frames with 4-second gaps. Verify pace is held (not interpolated across the gap).

5. **Dense vs sparse behavior**: Verify that with 1s sampling the pace updates immediately, while with 3s+ sampling it holds.
